Can We Share Device Telemetry Without Compromising Privacy?

We compared training-based and training-free differential privacy methods for generating synthetic relational data, benchmarked on 21 production-style SQL queries over Intel's Device Community Analytics corpus.

Why this matters

Device telemetry—hardware specs, usage patterns, network activity—helps product teams improve software, but it contains sensitive information about individuals. Releasing raw telemetry poses privacy risks. Differential privacy (DP) lets us publish synthetic data that preserves statistical utility while mathematically bounding the information leaked about any single record.

The core question: Which DP synthesis approach best preserves the utility of production SQL workloads on multi-table relational data?

  • Privacy risk: Raw telemetry can reveal device models, locations, browser habits, and usage patterns.
  • Solution: Generate synthetic data that mimics real distributions under formal privacy guarantees.
  • Trade-off: Privacy noise reduces utility. We measure this trade-off across four approaches and five query types.

Project scope

What we did

  • Implemented four DP synthesis pipelines (Wide DP-VAE, Per-table DP-SGD, MST, Private Evolution).
  • Designed a 21-query SQL benchmark covering aggregates, distributions, rankings, and pivots.
  • Evaluated all methods at matched privacy budget (ε = 4.0).
  • Analyzed failure modes: sparsity collapse, independence assumptions, LLM knowledge gaps.

What we did not do

  • Sweep over multiple ε values (fixed at 4.0 for fair comparison).
  • Implement relational DP methods that reason about foreign keys directly.
  • Fine-tune the LLM used in Private Evolution on domain-specific data.
  • Evaluate on datasets beyond the Intel DCA corpus.

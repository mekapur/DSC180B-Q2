Methods

Four approaches to differentially private synthetic data, evaluated on relational device telemetry.

What is differential privacy?

Differential privacy (DP) is a mathematical framework that quantifies privacy loss. It adds carefully calibrated noise so that an algorithm's output is nearly identical whether or not any single record is in the dataset.

Intuition: Imagine a census. DP ensures that published summary statistics would look almost the same whether or not your data was included. The privacy "cost" is measured by a parameter ε (epsilon)—smaller ε means stronger privacy but potentially lower utility.

We use (ε, δ)-differential privacy with ε = 4.0 and δ = 10−5, matched across all methods for fair comparison. This provides a moderate privacy guarantee suitable for internal analytics.

Formal definition (click to expand)

A randomized mechanism M satisfies (ε, δ)-DP if for all neighboring datasets D, D' (differing in one record) and all measurable sets S:

Pr[M(D) ∈ S] ≤ eε · Pr[M(D') ∈ S] + δ

In practice, DP-SGD tracks privacy loss via Rényi Differential Privacy (RDP) accountants and converts to (ε, δ) at the end of training.

Dataset: Intel Device Community Analytics

We use the Intel DCA corpus, a multi-table telemetry dataset with 8 relational tables covering hardware specifications, software configurations, network activity, power consumption, and user behavior across thousands of devices.

Tables are linked by a shared system_id foreign key. This relational structure is central to our evaluation—many benchmark queries require joining 2–3 tables.

Data handling

  • No data leakage: Synthetic data is generated from the full dataset; benchmark queries test statistical fidelity, not prediction.
  • No private data in outputs: All published results are aggregate statistics or synthetic data. No individual records are exposed.
  • Preprocessing: Continuous columns normalized to [0, 1]; categorical columns label-encoded. Missing values imputed per table.

Synthesis approaches

1. Wide-Table DP-VAE

Flatten all tables into a single wide table (left-join on system_id), then train a Variational Autoencoder (VAE) with DP-SGD. The VAE learns a compressed latent representation and generates synthetic records that preserve cross-table structure.

  • Strength: Preserves foreign-key relationships by design.
  • Weakness: Extreme sparsity (most columns are null after flattening) causes zero-inflation and mode collapse. Continuous metrics collapse to near-zero.

2. Per-Table DP-SGD

Train a separate VAE for each table independently using DP-SGD. This avoids the sparsity problem of wide-table flattening but breaks cross-table relationships.

  • Strength: High within-table fidelity; each model is simpler and faster to train.
  • Weakness: Foreign-key mismatches—a synthetic CPU record may reference a non-existent system.

3. MST (Maximum Spanning Tree)

A histogram-based baseline: estimate low-dimensional marginals under DP, build a dependency graph (Chow-Liu tree), and sample synthetic records that are consistent with the noisy marginals.

  • Strength: Interpretable, fast, works well for categorical data with clear dependencies.
  • Weakness: Exponential in table width; struggles with many continuous or high-cardinality columns.

4. Private Evolution

A training-free approach: use a large language model (LLM) to generate candidate synthetic records, then use the exponential mechanism (a DP algorithm) to select candidates that best match real data distributions.

  • Strength: No training required; can target specific query workloads.
  • Weakness: Limited by the LLM's training data—categories or values unseen by the model cannot be synthesized.

What the team built vs. reused

  • Built: The 21-query benchmark, the Per-table DP-SGD pipeline, the Private Evolution integration, and the evaluation harness.
